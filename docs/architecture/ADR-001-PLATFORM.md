# ADR-001: Выбор платформы — Open Journal Systems (OJS)

| Поле | Значение |
|------|----------|
| **Статус** | Принято |
| **Дата** | 2026-03-23 |
| **Решение принял** | ЦИИ совместно с заказчиком |

## Контекст

Необходимо выбрать платформу для сайта научного журнала «Вестник ЖезУ», которая:
- Обеспечивает полный цикл обработки рукописей (подача → рецензирование → публикация)
- Соответствует требованиям ККСОН к информационной открытости
- Поддерживает DOI, ORCID, многоязычность
- Поддерживается сообществом и имеет долгосрочную перспективу

## Рассмотренные варианты

### Вариант A: Open Journal Systems (OJS) 3.4.x
- **Плюсы**: бесплатная, открытый код, специально для научных журналов, «из коробки» поддерживает подачу/рецензирование/DOI/ORCID/многоязычность, рекомендована ЮНЕСКО, используется >25 000 журналов в мире, плагинная архитектура
- **Минусы**: PHP-стек (специфичный), кривая обучения для кастомизации тем, может быть тяжеловесна для простых задач

### Вариант B: Custom (Next.js / Django + собственная система)
- **Плюсы**: полный контроль над UI/UX, современный стек, гибкость
- **Минусы**: огромный объем разработки (workflow рецензирования, роли, уведомления, DOI, ORCID — всё с нуля), сроки 3–6 месяцев вместо 3–4 недель, требует постоянной поддержки

### Вариант C: WordPress + плагины
- **Плюсы**: простая установка, много тем
- **Минусы**: нет встроенного workflow рецензирования, плагины ненадежны, не соответствует стандартам научных журналов, не поддерживает CrossRef/DOI нативно

## Решение

**Выбран Вариант A: OJS 3.4.x**

Обоснование:
1. Прямо рекомендован в ТЗ заказчиком (пункт 4.1)
2. Покрывает 90% требований «из коробки» без программирования
3. Единственный вариант, укладывающийся в сроки (15–20 рабочих дней)
4. Стандарт де-факто для научных журналов в Казахстане и СНГ
5. Активное сообщество, регулярные обновления (PKP)

## Последствия

- Стек: PHP 8.1+, MySQL 8.0, Nginx
- Кастомизация дизайна через child theme (Smarty шаблоны + CSS)
- Необходимо изучить специфику OJS для разработки темы
- Развертывание через Docker (официальный образ PKP)
- Обновления OJS требуют тестирования совместимости с child theme

# ADR-002: Архитектура развертывания — Docker

| Поле | Значение |
|------|----------|
| **Статус** | Принято |
| **Дата** | 2026-03-23 |

## Контекст

Нужно определить способ развертывания OJS на сервере.

## Решение

Docker Compose с тремя сервисами:

```
┌─────────────────────────────────────────────┐
│                   Nginx                      │
│              (reverse proxy)                 │
│          SSL termination (443)               │
├─────────────────────────────────────────────┤
│                                              │
│  ┌──────────────┐     ┌──────────────────┐  │
│  │   OJS 3.4.x  │     │    MySQL 8.0     │  │
│  │   (PHP-FPM)  │────▶│                  │  │
│  │   Port 9000  │     │   Port 3306      │  │
│  └──────────────┘     └──────────────────┘  │
│         │                                    │
│         ▼                                    │
│  ┌──────────────┐                            │
│  │   Volumes    │                            │
│  │  - files/    │                            │
│  │  - public/   │                            │
│  │  - db_data/  │                            │
│  └──────────────┘                            │
└─────────────────────────────────────────────┘
```

### Обоснование:
- Изолированная среда, воспроизводимость
- Простота бэкапов (volumes)
- Легкость миграции между серверами
- Единообразие с существующей инфраструктурой ЦИИ

# ADR-003: Стратегия многоязычности

| Поле | Значение |
|------|----------|
| **Статус** | Принято |
| **Дата** | 2026-03-23 |

## Решение

- **Основной язык**: русский (ru_RU)
- **Дополнительные**: казахский (kk), английский (en_US)
- OJS нативно поддерживает многоязычность через локали
- Все метаданные статей вводятся на 3 языках при подаче
- UI интерфейс: используются стандартные переводы OJS + доработка казахской локали
- Переключатель языков в шапке сайта

### Казахская локаль

OJS не имеет полной казахской локали. Необходимо:
1. Скопировать русскую локаль как основу
2. Перевести ключевые строки интерфейса (меню, кнопки, формы)
3. Контрибьютить перевод в PKP (опционально)
